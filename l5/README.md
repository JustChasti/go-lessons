## 1. Срезы и циклы — паттерны обработки

**Накопление в новый срез.** Самый частый случай: пройти исходный срез, для каждого элемента решить, попадает он в результат или нет, и положить в новый срез через `append`. Так делается фильтрация, отображение (map), любая трансформация «по элементу». Исходный срез при этом не меняется. Если заранее ясно, что результат будет не больше исходного — есть смысл создать его через `make` с нулевой длиной и подходящей ёмкостью, чтобы избежать повторных перевыделений: каждый раз, когда внутреннему массиву не хватает места, рантайм выделяет новый, побольше, и копирует туда всё содержимое. Для маленьких срезов это незаметно, но в горячем коде имеет значение.

**In-place изменение.** Иногда нужно поменять сам срез, не создавая копию: убрать элемент по индексу, перевернуть, отсортировать. Здесь работают со срезом как с обычным массивом — через индексы. Стандартный приём для удаления элемента: склеить из двух частей через `append(s[:idx], s[idx+1:]...)` — это именно in-place, потому что под капотом результат пишется в тот же массив. Помни, что если у функции снаружи остаётся старый заголовок — он по-прежнему имеет старую длину и видит «лишний» элемент в хвосте. Поэтому такие операции возвращают новый срез: `s = remove(s, 2)`.

**Два указателя.** Универсальная техника: два индекса идут по срезу с разных позиций или с разной скоростью. Используется для поиска пары с заданной суммой, для удаления подряд идущих дубликатов, для проверки палиндрома. Главная идея — заменить вложенный цикл (квадратичный по времени) на один проход с двумя указателями (линейный). На лету ты учишься замечать ситуации, где это применимо.

**Накопительная сумма.** Идёшь по срезу, в новый срез кладёшь сумму всех увиденных значений. Удобный приём для задач, где потом нужно много раз считать сумму на отрезке: предсчитал префиксные суммы один раз — и любой запрос отвечается за константное время как разность.

**Рекурсия по срезу.** Срез часто можно описать рекурсивно: «результат для пустого среза — нейтральный элемент, для непустого — комбинация первого элемента и результата для хвоста». Это не самый эффективный способ обработки в Go (нет хвостовой оптимизации, плюс лишние аллокации на срезы-хвосты), зато прямой. Используется не как боевая техника, а как способ выразить идею решения, когда она напрашивается рекурсивно.

### Задачи

#### 1. `reverseSlice(nums []int) []int`

Вернуть новый срез с теми же элементами, что и `nums`, но в обратном порядке. Исходный срез не должен меняться. Создай результат через `make([]int, len(nums))` и заполни его одним проходом.

#### 2. `filterEvens(nums []int) []int`

Вернуть срез только из чётных элементов исходного, сохранив порядок. Это классическая фильтрация: создаёшь пустой результат, идёшь по исходному, чётные кладёшь в результат через `append`. Для пустого входа верни пустой срез (`[]int{}`), а не `nil` — это разные значения, и в тестах их нередко различают.

#### 3. `removeAt(nums []int, idx int) []int`

Вернуть новый срез без элемента с индексом `idx`. Если `idx` вне диапазона — верни копию исходного без изменений.

#### 4. `dedupSorted(nums []int) []int`

Из отсортированного по неубыванию среза удалить подряд идущие дубликаты, оставив каждое значение в одном экземпляре. Например, из `[1, 1, 2, 3, 3, 3, 4]` получится `[1, 2, 3, 4]`.

#### 5. `sumRec(nums []int) int`

Сумма элементов среза, реализованная через рекурсию. База: пустой срез — сумма 0. Шаг: первый элемент плюс сумма остального хвоста. Хвост получаешь через `nums[1:]` — это новый заголовок над тем же массивом, без копирования. Это не самый эффективный способ суммирования (для боевого кода — обычный цикл), но хорошая иллюстрация того, как срез разбирается рекурсивно.

#### 6. `maxRec(nums []int) int`

Максимум среза через рекурсию. Для пустого среза верни 0 (пограничный случай — оговорим). База: срез из одного элемента — он и есть максимум. Шаг: из первого элемента и максимума хвоста выбери больший. Здесь та же идея «голова + хвост», что и в `sumRec`, только вместо сложения — выбор максимального.

## 2. Многомерные срезы

Многомерный срез в Go — это срез срезов. `[][]int` — это срез, элементами которого являются срезы целых чисел.

### Зачем

Двумерные срезы появляются везде, где данные естественно ложатся на сетку: матрицы, шахматные доски и игровые поля, изображения как массив пикселей, таблицы (строки и столбцы), сжатые представления связности в графах. Любая задача в стиле «дано поле NxM, обходи его и считай что-то» — это двумерный срез.

### Объявление

Литералом — когда строки известны заранее: пишешь внешние фигурные скобки и внутри — каждую строку как отдельный срез. Через `make` — когда размеры известны во время выполнения: сначала создаёшь внешний срез нужной длины, а потом проходишь по нему циклом и каждой строке выделяешь отдельный внутренний срез. Это именно два шага, не один: `make([][]int, rows)` создаёт только внешний массив с `rows` пустыми (nil) внутренними срезами; чтобы появилась настоящая «таблица», каждый из них нужно инициализировать отдельно.

Из этого следует тонкий, но важный момент: внутренние срезы независимы друг от друга. Каждый — отдельная структура с собственными `len` и `cap`. Это значит, что разные строки могут иметь **разную длину**. Так получается то, что в литературе называют jagged array — «рваный массив». Когда тебе нужна именно матрица (все строки одной ширины), это твоя ответственность — следить за тем, чтобы они были одинаковыми. Никакой проверки от языка не будет.

### Обход

Стандартная схема — двойной цикл: внешний по строкам, внутренний по столбцам. Имя индексов обычно `i` и `j`, реже `r` и `c` — в зависимости от того, что подчёркиваешь. Для строки `matrix[i]` — это просто срез, и `len(matrix[i])` даёт ширину этой конкретной строки.

Внутри двойного цикла можно делать всё что угодно: считать сумму, искать максимум, заменять значения по правилу, печатать. Когда нужна не сумма всей матрицы, а суммы по строкам — внешний цикл копит результат в обычной переменной, и в конце каждой итерации внешнего цикла этот результат кладётся в накопительный срез.

### Модификация

Изменение элемента — простое: `matrix[i][j] = value`. Это работает потому, что слева у нас «строка», и из неё через индекс достаётся изменяемая ячейка

### Игровые поля и карты

Один из самых наглядных примеров двумерного среза — текстовая карта: `[][]rune`, где каждая ячейка — это символ, например `'#'` для стены и `'.'` для прохода. Двойной цикл идёт по карте и принимает решения по содержимому. Это базовая модель для многих игровых задач: от пути в лабиринте до симуляций. В файле есть короткий пример — карта 5x10 со стенами и счёт открытых клеток.

## 3. Интерфейсы и видимость

В Go нет классов, нет наследования, нет приватных полей в смысле, привычном по другим языкам. Но идея инкапсуляции — спрятать детали и оставить наружу только нужное — реализуется двумя механизмами: видимостью на уровне пакета и интерфейсами.

### Видимость

В Go видимость определяется одной простой правилом: имя, начинающееся с заглавной буквы, экспортируется из пакета — оно видно из других пакетов; имя, начинающееся со строчной, видно только внутри своего пакета. Это касается всего: типов, полей структур, методов, функций, констант. Тип `User` экспортируется, тип `user` — нет. Поле `Name` видно снаружи, поле `password` — только в пакете, где `User` объявлен.

В отличие от других языков, у Go нет понятия «приватный для объекта»: все методы и поля одного типа видны из любого кода в том же пакете. Граница приватности — это пакет, а не структура. На практике этого хватает: пакет — естественная единица модульности, и весь код, тесно связанный с типом, обычно живёт рядом с ним.

Когда снаружи доступно только Public-API, а детали реализации спрятаны — это и есть инкапсуляция. Поле `password` в структуре `User` хранится внутри, но никакой код вне пакета его не прочитает напрямую. Чтобы установить пароль, надо вызвать метод `SetPassword` (он, например, хеширует значение перед сохранением). Чтобы проверить пароль — `CheckPassword`. Снаружи у тебя нет возможности случайно (или умышленно) положить в это поле что попало или прочитать сырое содержимое — это и есть инкапсуляция в Go-смысле.

### Методы

Метод — это функция с получателем (receiver), привязанная к типу. Объявление выглядит как обычная функция, но перед именем стоит `(u User)` или `(u *User)` — это и есть получатель. Внутри метода `u` — это значение или указатель на значение, на котором вызвали метод.

Получатель по значению или по указателю — выбирается по тем же двум причинам, что и в обычных функциях. По указателю — если метод должен **менять** состояние или если структура большая и не хочется её копировать. По значению — если метод просто читает поля и структура небольшая. На практике в одном типе методы лучше делать однородными: либо все по указателю, либо все по значению. Смешивание усложняет понимание и иногда приводит к нюансам с интерфейсами.

### Интерфейс

Интерфейс — это набор методов. Он описывает поведение, а не данные. Объявление `type Animal interface { Speak() string; Name() string }` означает: «всё, что умеет Speak и Name — это Animal». Никаких полей, никакой реализации внутри.

Главная особенность интерфейсов в Go — **неявное удовлетворение** (implicit satisfaction). Тебе не нужно где-то писать «тип `Dog` реализует интерфейс `Animal`». Ты просто пишешь у `Dog` методы `Speak()` и `Name()`, и с этого момента `Dog` автоматически удовлетворяет `Animal`. Компилятор сам это отслеживает в момент, когда ты пытаешься положить `Dog` в переменную типа `Animal` — или сообщит об ошибке, если каких-то методов не хватает.

Это даёт два важных свойства. Первое — слабая связанность: тип, реализующий интерфейс, ничего не знает о существовании самого интерфейса. Можно описать новый интерфейс под старый тип задним числом — и он сразу заработает. Второе — простота: нет церемоний, нет аннотаций, нет магических деклараций. Если тип удовлетворяет — он удовлетворяет.

### Полиморфизм через срез интерфейсов

Самый частый сценарий использования интерфейсов: складываем разные конкретные типы (`Dog`, `Cat`, `Cow`) в один срез типа `[]Animal` и в цикле вызываем общий метод (`a.Speak()`). Каждое значение «помнит» свой настоящий тип, и при вызове метода выполняется именно его реализация. Снаружи код этого не различает — он работает с интерфейсом.

Это позволяет писать функции, которые принимают абстрактное поведение, а не конкретный тип. Стандартная библиотека построена именно так: `io.Reader` — это «всё, что умеет читать байты», `io.Writer` — «всё, что умеет писать», и тонна функций работает с этими интерфейсами, не зная, что под ними — файл, сеть, буфер в памяти или даже сжатый поток.

## 4. Обработка ошибок

В Go нет исключений. Это не упущение, а сознательное архитектурное решение. Ошибка в Go — обычное значение, которое функция возвращает наряду с результатом. Тот, кто вызвал функцию, тот и решает, что делать с ошибкой: проверить, обработать, передать дальше. Ничего «не вылетает» само — управление не переходит куда-то неожиданно через try/catch, всё видно прямо на месте.

### error как интерфейс

Тип `error` — это встроенный интерфейс с одним методом: `Error() string`. Любой тип, у которого есть метод `Error() string`, удовлетворяет `error`. Это связь с предыдущей темой: обработка ошибок в Go сделана прямо поверх механики интерфейсов.

Самый простой способ создать ошибку — `errors.New("not found")` или `fmt.Errorf("user %d not found", id)`. Оба возвращают `error`. Первый — для константных сообщений, второй — для сообщений с подстановкой значений.

### Возврат двух значений

Канонический паттерн в Go — функция, которая может ошибиться, возвращает два значения: результат и ошибку. Если ошибки нет — `error` равен `nil`, и результат можно использовать. Если ошибка есть — `error` ненулевой, а результат, как правило, лежит в нулевом значении (но полагаться на это нельзя — всегда сначала проверяем ошибку).

Поэтому в каждой второй строке Go-кода ты будешь видеть конструкцию `if err != nil { return ..., err }`. Это и есть «многословность Go в обработке ошибок», на которую часто жалуются — и в которой одновременно видна сильная сторона: ошибка не может быть проигнорирована тихо, она физически торчит в коде.

### Sentinel-ошибки и errors.Is

Иногда вызывающему коду важно различать **виды** ошибок: «файла нет» и «нет прав на чтение» — это разные ситуации. Самый простой способ — определить ошибку как переменную пакета: `var ErrNotFound = errors.New("not found")`. Когда функция возвращает эту конкретную ошибку, вызывающий код может сравнить её через `errors.Is(err, ErrNotFound)` и обработать особым образом.

Прямое сравнение `err == ErrNotFound` тоже работает, но `errors.Is` умеет проходить через обёртки (см. ниже про %w) — поэтому в современных кодовых базах используют именно его.

### Обёртывание через fmt.Errorf и %w

Часто хочется добавить к ошибке контекст: какой файл не открылся, какой запрос упал. Делается это через `fmt.Errorf("read config: %w", err)`. Глагол `%w` (wrap) — специальный: он прячет исходную ошибку внутрь новой, но не теряет её. Снаружи мы видим строку «read config: open file.json: no such file», а под капотом исходный `os.ErrNotExist` всё ещё там, и `errors.Is(err, os.ErrNotExist)` корректно ответит true.

Так строятся «цепочки ошибок» — каждый уровень кода добавляет свой слой контекста, не теряя исходную причину. Это проще, чем стектрейс, и ближе к тому, как реально устроен путь данных через систему.

### Свой тип ошибки

Когда нужно передавать вместе с ошибкой какие-то поля (код HTTP, имя файла, индекс строки в CSV) — определяют свой тип. Это обычная структура с полями плюс метод `Error() string`. После этого она удовлетворяет интерфейсу `error` и ходит как любая другая ошибка. Извлечь из неё конкретный тип на стороне вызывающего можно через `errors.As`: он принимает указатель на переменную нужного типа и заполняет её, если в цепочке ошибок такая обнаружится.

### panic и recover — отдельная история

В Go есть `panic` и `recover` — это механизм, похожий внешне на исключения, но используется для другого. `panic` — это «всё пропало, программа в неконсистентном состоянии, дальше работать нельзя»: индекс вне диапазона, разыменование nil, явная проверка инварианта. По умолчанию paник раскручивает стек и убивает программу. `recover` позволяет в `defer`-функции перехватить panic и не дать программе упасть.

Важно: panic/recover **не используется** для обычной бизнес-логики. Если функция может не найти пользователя — это `error`, а не paник. Panic применяется только к ситуациям, в которых программа не может разумно продолжать работу, — и даже тогда `recover` обычно применяется на верхнем уровне (например, в HTTP-обработчиках), чтобы один сломавшийся запрос не уронил весь сервер.

## 5. Домашка: две задачи с двумерными массивами

Реализации лежат в `resolve/05_homework.go` — посмотришь после того, как сам напишешь.

#### 1. `transpose(matrix [][]int) [][]int`

Транспонировать матрицу: то, что было строкой, должно стать столбцом, и наоборот. Если на входе матрица размера M×N (M строк, N столбцов), на выходе будет N×M.

Например, из

```
1 2 3
4 5 6
```

получится

```
1 4
2 5
3 6
```

Строка с индексом `i` и столбец `j` исходной матрицы становятся строкой `j` и столбцом `i` результата. Создавай новую матрицу — исходную не трогай. Для пустой матрицы (`len(matrix) == 0`) верни пустой срез.

#### 2. `spiralOrder(matrix [][]int) []int`

Обойти прямоугольную матрицу по спирали — от левого верхнего угла направо, затем вниз по правому краю, затем влево по нижней стороне, затем вверх по левому краю, и так, по сужающимся кольцам, до центра. Результат — одномерный срез всех элементов в порядке обхода.

Например, для

```
1 2 3
4 5 6
7 8 9
```

ответ — `[1, 2, 3, 6, 9, 8, 7, 4, 5]`.

Стандартный приём — четыре границы (`top`, `bottom`, `left`, `right`), которые сужаются после прохода каждой стороны. Внешний цикл идёт пока границы не схлопнулись. Внутри — четыре прохода: направо по верхней строке, вниз по правому столбцу, налево по нижней строке (если она ещё существует), вверх по левому столбцу (если он ещё существует).

Тонкость: после первой стороны нужно проверять, не схлопнулись ли границы — иначе на узких матрицах (одна строка или один столбец) можно случайно пройти элементы повторно.
